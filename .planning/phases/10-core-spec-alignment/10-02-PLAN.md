**Step 1: Update StepTextData interface** in `step-data.ts`:
- Add `input_data` field as an alias: `input_data: string | null;`
- Keep `data` field for backward compatibility with any code that sets it directly
- Both `data` and `input_data` can coexist on the interface; serialization will use `input_data`
- Actually, the cleaner approach per user decision "serialization-time mapping": keep `data` as the field name on the interface, and map it to `input_data` during serialization. So do NOT change `step-data.ts` interface fields.

**CORRECTION**: Do NOT modify `step-data.ts`. The user decision says "internal model field names preserved; serialization-time mapping for backward compatibility." The `StepTextData.data` field stays as `data`. The serialization helper will map it to `input_data` when writing to file.

**Step 2: Create serialization helper** in `report-reporter.ts`:

Add a private method `serializeResultForReport(result: TestResultType): Record<string, unknown>` that transforms the internal model to spec-compliant JSON shape. This method creates a plain object (not a class instance) that will be passed to `writeTestResult` via the formatter.

The transformation must:

a) **testops_id -> testops_ids** (RSLT-01):
   - If `result.testops_id` is `null` -> `testops_ids: null`
   - If `result.testops_id` is a number -> `testops_ids: [result.testops_id]`
   - If `result.testops_id` is already an array -> `testops_ids: result.testops_id`
   - Remove `testops_id` key from output

b) **group_params -> param_groups** (RSLT-02):
   - `result.group_params` is `Record<string, string>` internally
   - Spec wants `param_groups: array<array<string>>`
   - Use same transformation as `clientV2.ts` `transformGroupParams()`: convert Record keys to `string[][]`
   - If `group_params` is empty -> `param_groups: []`
   - Remove `group_params` key from output

c) **Step data.data -> data.input_data** (STEP-01):
   - For each step (recursively), if step_type is 'text' and data has `data` field:
     - Rename `data` property to `input_data` in serialized output
   - Create helper `serializeSteps(steps: TestStepType[]): Record<string, unknown>[]`

d) **Step attachments -> execution.attachments** (STEP-02):
   - For each step, move `attachments` from top-level into `execution.attachments`
   - Serialized step should NOT have top-level `attachments` array
   - The `execution` object should include `attachments` array

e) **Exclude attachment size** (XTRA-01):
   - When serializing attachments (both result-level and step-level), omit the `size` field
   - Also omit `content` field (it should not be in file output, it's runtime-only)
   - Create helper `serializeAttachment(att: Attachment): Record<string, unknown>` that includes only: `id`, `file_name`, `mime_type`, `file_path`

**Step 3: Integrate serialization into sendResults()**:

Modify `sendResults()` in `report-reporter.ts`:
- After all the existing logic (attachment writing, step attachment copying, run_id, relations), before calling `this.writer.writeTestResult(result)`:
- Create the serialized object: `const serialized = this.serializeResultForReport(result)`
- Pass serialized to writer: `await this.writer.writeTestResult(serialized as any)`
- The writer's `writeTestResult` signature accepts `TestResultType` but since it just calls `formatter.format(result)` which takes `unknown`, passing a plain object works fine.

IMPORTANT: This approach means the writer receives a spec-compliant plain object instead of the TestResultType class instance. The writer just passes it through to `JSON.stringify`, so this works without changing the writer interface.

Also exclude these internal-only fields from the serialized output:
- `testops_project_mapping` (internal multi-project mapping, not in spec)
- `preparedAttachments` (internal, not in spec)
- `run_id` (not in result spec - it's only used internally)
- `author` (not in result spec)

The serialized result should have exactly these fields per result.yaml:
- `id`, `title`, `signature`, `execution`, `fields`, `attachments`, `steps`, `params`, `param_groups`, `testops_ids`, `relations`, `muted`, `message`
